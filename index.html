Running LLM Inference on 4× Tesla V100 — Bare Metal to Production

AWS p3.8xlarge (4× V100) costs ~$8,800/month. We own the hardware. Here's the full stack: K3s, vLLM, FastAPI load-balancer, nginx API key auth — serving 10,000+ daily requests at sub-400ms P95.

The Why: Cloud GPUs Are Expensive

AWS p3.8xlarge (4× V100, 16GB each) costs $12.24/hour — $8,800/month for 24/7 availability. We owned the hardware. Colocation + electricity brought the effective monthly cost to ~$180. The decision was straightforward.

OS & CUDA Setup

Ubuntu 22.04 LTS — not 24.04. CUDA driver stability and nvidia-container-toolkit package availability were better tested on 22.04 at the time of setup. We installed CUDA 12.x and the NVIDIA Container Toolkit to allow Docker and K3s pods to access the GPUs.

distribution=$(. /etc/os-release; echo $ID$VERSION_ID)
curl -s -L https://nvidia.github.io/libnvidia-container/gpgkey | apt-key add -
apt install nvidia-cuda-toolkit nvidia-container-toolkit
systemctl restart docker

K3s: Lightweight Kubernetes on One Server

Full Kubernetes felt like overkill for a single physical node. K3s runs the same Kubernetes API surface at a fraction of the resource overhead. We deployed:

  • nginx Ingress Controller — external traffic routing
  • cert-manager + Let's Encrypt — automatic TLS, zero manual certificate management
  • CoreDNS — internal service discovery across LXD containers and K3s pods
  • NVIDIA Device Plugin — exposes nvidia.com/gpu as a schedulable resource

Each vLLM worker pod requests exactly nvidia.com/gpu: 1, so K3s handles GPU allocation automatically.

vLLM + Qwen2.5-Coder-32B

Qwen2.5-Coder-32B needs ~60GB VRAM in BF16. With 4× 16GB = 64GB total, there's just enough headroom for KV cache. tensor_parallel_size=4 splits the model across all four V100s using NVLink-style communication over PCIe.

from vllm import LLM, SamplingParams

llm = LLM(
    model="Qwen/Qwen2.5-Coder-32B-Instruct",
    tensor_parallel_size=4,
    gpu_memory_utilization=0.91,
    max_model_len=8192
)

Keep gpu_memory_utilization at 0.90–0.92. Going higher causes OOM crashes during KV cache spikes on long-context requests.

FastAPI Load-Balancing Router

Each GPU worker runs as an independent vLLM OpenAI-compatible server on a different port. A FastAPI router round-robins across them with health checks — no traffic goes to a worker still loading the model (cold start is 3–4 minutes).

worker_pool = cycle(["http://llm-worker-0:8000",
                     "http://llm-worker-1:8000",
                     "http://llm-worker-2:8000",
                     "http://llm-worker-3:8000"])

@app.post("/v1/completions")
async def complete(req: Request, api_key: str = Depends(verify_key)):
    target = next(worker_pool)
    async with httpx.AsyncClient(timeout=120) as client:
        r = await client.post(f"{target}/v1/completions",
                              content=await req.body(),
                              headers=req.headers)
    return Response(content=r.content, media_type="application/json")

nginx: API Key Auth + Rate Limiting

upstream llm_upstream {
    server llm-router:8080;
}
location /v1/ {
    auth_request     /auth;
    proxy_pass       http://llm_upstream;
    limit_req        zone=per_key burst=10 nodelay;
    proxy_read_timeout 120s;
}
Results: 10,000+ daily completions served · Cold start ~3.5 min · P95 token generation ~400ms for 256-token output · Monthly infra cost ~$180 vs ~$8,800 cloud equivalent

Key Takeaways

  • K3s + NVIDIA device plugin works on single-node bare metal with zero special config
  • tensor_parallel_size must match physical GPU count for V100 PCIe topology
  • Health-check poll the worker before routing — model load takes time and silently fails requests otherwise
  • LXD containers for dev isolation + K3s for production on the same host works well; use CoreDNS to bridge them

Stopping a Live Stripe Carding Attack — Zero Fraud Post-Deployment

400+ failed charges in two hours, all $0.99, all from Eastern Europe. We went from 12% fraud rate to 0.02% with three layers: Stripe Radar rules, Redis rate limiting per IP/BIN, and HMAC inter-service authentication.

What is a Carding Attack?

Attackers buy lists of stolen card numbers and run automated scripts to test them in small amounts — usually $0.50–$1.00 — against real merchant endpoints. Cards that succeed are "live" and get sold at a premium. You only notice when Stripe emails you about elevated dispute rates.

How We Found Out

Our Stripe dashboard showed 400+ failed charges in a 2-hour window: all $0.99, multiple IPs from Eastern Europe, all hitting the same /api/checkout endpoint, all with different card numbers. Classic carding signature.

Layer 1: Stripe Radar Rules

Radar's rule DSL lets you block or review charges based on risk signals. We added three rules immediately:

# Block high-risk country + low amount
block if: :ip_country: in ('RU', 'UA', 'RO', 'BG', 'BY') and :amount_in_cents: < 200

# Block velocity per IP
block if: charge_count_per_ip_hourly > 3

# Review first-time international cards
review if: :is_new_card: = true and :card_country: != 'PK' and :amount_in_cents: < 500

This cut ~60% of the attack volume at Stripe's edge before charges were even attempted.

Layer 2: Redis Rate Limiting Per IP + BIN

Radar handles Stripe-side, but we needed to stop requests reaching our API at all. Token bucket per IP + card BIN prefix (first 6 digits — identifies the issuing bank):

async def check_rate_limit(ip: str, card_bin: str) -> bool:
    key = f"checkout:{ip}:{card_bin[:6]}"
    count = await redis.incr(key)
    if count == 1:
        await redis.expire(key, 3600)  # 1-hour window
    if count > 3:
        raise HTTPException(429, "Too many payment attempts")
    return True

Rate limiting on BIN prefix — not just full card number — catches attackers cycling through different cards from the same issuing bank, which is a common evasion pattern.

Layer 3: HMAC Inter-Service Authentication

The payment service should only be callable from our own backend — not directly from any external client. We added HMAC-SHA256 request signing between services:

def sign_request(payload: bytes, secret: str) -> str:
    return hmac.new(
        secret.encode(), payload, hashlib.sha256
    ).hexdigest()

# Verification middleware
async def verify_hmac(request: Request):
    sig = request.headers.get("X-Signature")
    body = await request.body()
    expected = sign_request(body, SHARED_SECRET)
    if not hmac.compare_digest(sig or "", expected):
        raise HTTPException(401, "Invalid signature")

Any request without a valid HMAC from a known internal service is rejected before it touches the payment logic or Stripe.

Result: Fraud rate 12% → 0.02% · Zero fraudulent transactions post-deployment · No legitimate customer blocked · Full mitigation deployed in under 4 hours

What We Learned

  • Configure Radar rules proactively — reactive setup during an active attack is stressful
  • Rate limit on BIN prefix (first 6 digits), not just full card — attackers cycle cards from the same bank
  • HMAC between your own services is cheap to add and eliminates an entire attack surface entirely
  • Keep a Stripe webhook endpoint logging all events — the audit trail is invaluable for post-incident analysis

gRPC vs REST in Production — How We Cut Internal API Overhead by 40%

Six microservices, all talking REST over HTTP/1.1. A single agent task triggered 5 synchronous cross-service calls. Here's what we measured, how we migrated, and when gRPC is — and isn't — the right call.

The Problem

The AI agent platform had 6 services: API Gateway, Agent Orchestrator, Task Queue, LLM Router, User Service, and Billing. All REST over HTTP/1.1. A single agent task execution path: Gateway → Orchestrator → LLM Router → Task Queue → back to Orchestrator. Five round-trips, each with JSON serialisation overhead, connection setup, and no streaming.

Why gRPC?

  • Protocol Buffers: binary serialisation — ~3–10× smaller payloads than JSON, faster to serialise/deserialise
  • HTTP/2: connection multiplexing — one persistent connection, no per-request TCP overhead
  • Server-side streaming: LLM token streaming over gRPC is natural; over REST it requires SSE or websockets
  • Typed contracts: .proto files as source of truth — no more runtime JSON field mismatches

The .proto Contract

syntax = "proto3";

service AgentService {
  rpc ExecuteTask(TaskRequest) returns (TaskResponse);
  rpc StreamOutput(TaskRequest) returns (stream OutputChunk);
}

message TaskRequest {
  string task_id    = 1;
  string agent_id   = 2;
  string prompt     = 3;
  map<string, string> metadata = 4;
}

message OutputChunk {
  string content    = 1;
  bool   is_final   = 2;
}

Migration Strategy — No Big-Bang Rewrite

We didn't rewrite everything at once. We ran gRPC alongside REST: added gRPC servers to each service, moved service-to-service calls first, left the public-facing API as REST. Each service could still be tested via REST during the transition. Migration took two weeks with zero downtime.

Python gRPC with asyncio

import grpc
from grpc.aio import Channel

class AgentClient:
    def __init__(self, channel: Channel):
        self.stub = AgentServiceStub(channel)

    async def execute(self, task: TaskRequest) -> TaskResponse:
        return await self.stub.ExecuteTask(task, timeout=30)

    async def stream(self, task: TaskRequest):
        async for chunk in self.stub.StreamOutput(task):
            yield chunk.content
Numbers: Internal latency avg 12ms → 7ms (−42%) · Payload size avg 1.4KB → 340 bytes (−76%) · P95 end-to-end agent task 180ms → 108ms · Connection overhead eliminated via HTTP/2 multiplexing

When NOT to Use gRPC

  • Public APIs: browsers can't use gRPC natively (grpc-web is a workaround but adds complexity)
  • Simple infrequent CRUD: proto compilation overhead isn't worth it for 10 req/day endpoints
  • Mixed-language teams new to proto: debugging binary protocol traffic is harder than JSON — invest in tooling first

Takeaway

For service-to-service calls within the same infrastructure, gRPC is almost always the right choice. Keep REST for public APIs and external partner integrations where client diversity matters.

Zero-Trust Secrets with HashiCorp Vault — GDPR-Compliant on Bare Metal

Secrets in .env files, CI environment variables, and SSH keys in repos — the flat permission model that most teams live with. Here's how we replaced all of it with AppRole auth, policy-based access, and on-prem secret residency for GDPR compliance.

The Problem with .env Files

Every project has them. The issues compound quietly: .env accidentally committed once (it happens), CI environment variables visible to every engineer in the org, database passwords copy-pasted into Slack for "quick sharing", SSH keys that were never rotated after an engineer left. This isn't zero-trust — it's a flat, unaudited permission model.

Why Self-Hosted Vault Over Cloud Secrets Managers?

AWS Secrets Manager and GCP Secret Manager are excellent products. But our GDPR data residency requirement meant customer-related secrets couldn't live in AWS us-east-1 or route through US infrastructure. Vault in an LXD container on our German-hosted bare-metal server solved this: secrets never leave the jurisdiction, audit logs are on-prem, and there's no dependency on a cloud provider's API uptime.

AppRole Authentication

AppRole is designed for machine-to-machine auth. Each service gets a role_id (public, like a username) and a secret_id (private, short-lived — like a one-time password):

vault write auth/approle/role/api-service \
    token_policies="api-service-policy" \
    secret_id_ttl=1h \
    token_ttl=30m \
    token_max_ttl=2h

The CI/CD pipeline fetches a fresh secret_id per run. It expires in 1 hour. Even if it leaks from a build log, the attack window is narrow and auditable.

Policy-Based Access Control — Least Privilege

Each service can only read its own secrets. The LLM service cannot read the billing service's Stripe keys. Period.

# api-service-policy.hcl
path "secret/data/api-service/*" {
  capabilities = ["read"]
}

# billing-service-policy.hcl
path "secret/data/billing/*" {
  capabilities = ["read"]
}
# No cross-service reads — ever.

CI/CD Integration — GitHub Actions

Secrets are fetched at runtime, not baked into the image or stored in GitHub Secrets:

- name: Fetch secrets from Vault
  run: |
    SECRET_ID=$(vault write -f \
      auth/approle/role/sonar/secret-id \
      -format=json | jq -r '.data.secret_id')

    TOKEN=$(vault write auth/approle/login \
      role_id=${{ secrets.ROLE_ID }} \
      secret_id=$SECRET_ID \
      -format=json | jq -r '.auth.client_token')

    echo "SONAR_TOKEN=$(vault kv get \
      -field=token secret/sonar)" >> $GITHUB_ENV

GDPR: Why Data Residency Matters for Secrets

GDPR Article 32 requires appropriate technical measures for the security of processing. If your encryption keys and database credentials route through a US cloud provider's API to be fetched, you've introduced a cross-border data flow risk. On-prem Vault eliminates this entirely — the secret retrieval call never leaves your server rack.

Result: Zero .env files in production · Secret rotation = one Vault write (services pick it up on next token refresh) · Every secret access is audited with timestamp, service identity, and IP · Full GDPR Article 32 compliance documented

Key Takeaways

  • Start with AppRole + short-lived secret_ids — it's the right auth method for CI/CD and services
  • Write policies before you write secrets — least privilege from day one
  • Enable Vault audit logging immediately; it's the paper trail that matters during incidents
  • If GDPR applies to you, self-hosted Vault is a legitimate compliance argument — document it explicitly

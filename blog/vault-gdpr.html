Zero-Trust Secrets with HashiCorp Vault — GDPR-Compliant on Bare Metal

Dec 2025·8 min read·Talha Zafar

Secrets in .env files, CI environment variables, and SSH keys copied into Slack — the flat permission model that most teams live with. Here's how we replaced all of it with AppRole auth, policy-based access, and on-prem secret residency for full GDPR Article 32 compliance.

The Problem with .env Files

Every project has them. The issues compound quietly: .env accidentally committed once (it happens), CI environment variables visible to every engineer in the org, database passwords copy-pasted into Slack for "quick sharing", SSH keys never rotated after an engineer left.

The incident that triggered this for us: a developer left the company and their SSH key — shared "temporarily" for an urgent deployment — was still active six months later. We had no audit trail of what it had accessed. That was the moment we knew we needed Vault.

Why Self-Hosted Vault Over Cloud Secrets Managers?

AWS Secrets Manager and GCP Secret Manager are excellent. But our GDPR data residency requirement was specific: customer-related encryption keys and database credentials must not route through US infrastructure. Every AWS Secrets Manager API call routes through Amazon's global infrastructure regardless of your region setting.

Vault in an LXD container on our German-hosted bare-metal server solved this entirely. The secret retrieval call never leaves the server rack. Every access is logged locally. We can produce a complete audit trail for any GDPR Article 32 audit.

Vault in LXD: The Setup

# Create isolated LXD container
lxc launch ubuntu:22.04 vault-server
lxc exec vault-server -- bash

# Install Vault inside container
wget -O- https://apt.releases.hashicorp.com/gpg | gpg --dearmor \
  -o /usr/share/keyrings/hashicorp.gpg
apt install vault

# /etc/vault.d/vault.hcl
storage "file" {
  path = "/opt/vault/data"
}
listener "tcp" {
  address     = "0.0.0.0:8200"
  tls_disable = 1  # TLS terminated at nginx reverse proxy
}
api_addr = "https://vault.internal.yourdomain.com"

AppRole Authentication

AppRole is built for machine-to-machine auth. Each service gets a role_id (public, like a username) and a secret_id (private, short-lived). The key property: secret_id expires after a configurable TTL — even if it leaks from a CI log, the attack window is narrow and bounded.

# Enable AppRole auth method
vault auth enable approle

# Create a role per service
vault write auth/approle/role/api-service \
    token_policies="api-service-policy" \
    secret_id_ttl=1h \
    token_ttl=30m \
    token_max_ttl=2h \
    bind_secret_id=true

# role_id is non-sensitive — store in CI as a plain variable
vault read auth/approle/role/api-service/role-id

Policy-Based Access Control — Least Privilege

Each service reads only its own secrets. The LLM service cannot read the billing Stripe keys. The billing service cannot read LLM router tokens. Enforced at the Vault policy level — not by convention.

# api-service-policy.hcl
path "secret/data/api-service/*" {
  capabilities = ["read", "list"]
}
# Explicit deny — nothing implicit
path "secret/data/billing/*" {
  capabilities = ["deny"]
}
path "secret/data/llm-router/*" {
  capabilities = ["deny"]
}

vault policy write api-service api-service-policy.hcl

CI/CD Integration — GitHub Actions

Secrets are fetched at runtime, never baked into images or stored in GitHub Secrets. Each CI run gets a fresh secret_id that expires in 1 hour.

- name: Fetch secrets from Vault
  env:
    VAULT_ADDR: ${{ secrets.VAULT_ADDR }}
    ROLE_ID: ${{ secrets.ROLE_ID }}   # Non-sensitive: like a username
  run: |
    # Fresh secret_id per run — expires in 1h
    SECRET_ID=$(vault write -f \
      auth/approle/role/sonar/secret-id \
      -format=json | jq -r '.data.secret_id')

    # Exchange for a short-lived Vault token (30min)
    VAULT_TOKEN=$(vault write auth/approle/login \
      role_id=$ROLE_ID \
      secret_id=$SECRET_ID \
      -format=json | jq -r '.auth.client_token')

    # Fetch the actual secret
    echo "SONAR_TOKEN=$(VAULT_TOKEN=$VAULT_TOKEN \
      vault kv get -field=token secret/sonar)" >> $GITHUB_ENV

If the secret_id leaks from a CI log, it's only usable for 1 hour and only grants access to what that role's policy allows — not the entire secrets store.

Dynamic Credentials for SonarQube

For SonarQube in CI, we went further: instead of a static token in Vault, we configured Vault to generate a unique token per CI run that's automatically revoked after the pipeline completes. Each pipeline run gets credentials that have never existed before and never will again.

GDPR Article 32: Why On-Prem Matters

GDPR Article 32 requires appropriate technical measures for the security of processing. If your encryption keys route through a US cloud provider's API, you've created a cross-border data flow under GDPR Chapter V that must be documented and justified.

On-prem Vault eliminates this entirely. Secret retrieval never leaves the server. Every access is logged locally with timestamp, service identity, and IP. You produce the complete audit trail from your own server — no dependency on a cloud provider's log export API.

Result: Zero .env files in production · Secret rotation = one Vault write, services pick it up on next token refresh · Every access audited with timestamp + service identity · GDPR Article 32 compliance documented with on-prem audit logs · Vault uptime: 99.8%

Key Takeaways

  • Start with AppRole + short-lived secret_ids — the right auth method for CI/CD from day one, not something to retrofit later
  • Write policies before you write secrets — least privilege from the beginning is far easier than retrofitting it across a live system
  • Enable audit logging immediately: vault audit enable file file_path=/var/log/vault_audit.log — this log is the paper trail that matters during incidents
  • If GDPR applies, self-hosted Vault is a legitimate and documentable compliance argument under Article 32 — write it into your security documentation explicitly
  • LXD for Vault isolation is a solid pattern — full network namespace, filesystem, and resource limit isolation without VM overhead

Zero-Trust Secrets with HashiCorp Vault — GDPR-Compliant on Bare Metal

Dec 2025·8 min read·Talha Zafar

We found out a developer who'd left the company six months earlier still had an active SSH key. Not just inactive — still authenticating. We had no audit trail of what it had touched. That was the conversation that ended with me setting up HashiCorp Vault the following week.

The .env file problem nobody wants to admit

Most teams have the same pattern. There's a .env.example in the repo, an actual .env that's gitignored (usually), and a Notion doc somewhere with the "real" values that needs to be updated whenever something changes. New developers get onboarded via Slack DMs with database passwords. Secrets get rotated when someone remembers or after an incident. Nobody actually knows who has access to what.

The SSH key incident made this concrete. A developer had been given temporary SSH access for an urgent deployment. "Temporary" turned into six months because there was no process for revoking it. By the time we noticed, he'd been at another company for half a year. We didn't know if the key had been used since he left — we had no log. That kind of uncertainty is exactly what GDPR's Article 32 is written to prevent.

Why not just use AWS Secrets Manager?

AWS Secrets Manager is well-designed and I'd recommend it to most teams. But we had a specific constraint: our customers are European businesses, and our DPA explicitly says customer-related encryption keys and database credentials must not route through US infrastructure.

Here's the thing about AWS Secrets Manager that surprised me when I looked into it: even if your AWS region is eu-central-1, the Secrets Manager API calls still route through Amazon's global infrastructure. The data is stored in your region, but the control plane — the API you call to retrieve secrets — is not purely regional. For a strict GDPR data residency interpretation, that's a problem you have to document and justify.

Vault in an LXD container on our German bare-metal server sidesteps this entirely. Secret retrieval never leaves the rack. Every access is logged locally. We can produce a complete audit trail from our own server, no dependency on a cloud provider's log export API.

Vault in LXD: the actual setup

I put Vault in an LXD container for isolation — separate network namespace, separate filesystem, resource limits. It runs alongside the other LXD containers on the same server but can't reach them unless explicitly allowed. Vault's own documentation recommends a dedicated machine; LXD is a reasonable compromise when you don't have one.

# Create isolated LXD container
lxc launch ubuntu:22.04 vault-server
lxc exec vault-server -- bash

# Install Vault inside container
wget -O- https://apt.releases.hashicorp.com/gpg | gpg --dearmor \
  -o /usr/share/keyrings/hashicorp.gpg
apt install vault

# /etc/vault.d/vault.hcl
storage "file" {
  path = "/opt/vault/data"
}
listener "tcp" {
  address     = "0.0.0.0:8200"
  tls_disable = 1  # TLS terminated at nginx reverse proxy
}
api_addr = "https://vault.internal.yourdomain.com"

TLS is disabled in Vault's config because nginx handles it at the proxy level — Let's Encrypt cert, automatic renewal via cert-manager. This is a common and clean pattern. Just don't expose Vault's 8200 port directly — nginx is the only thing that should talk to it from outside the container.

AppRole auth — built for machines, not humans

Vault has several authentication methods. AppRole is the one designed specifically for machine-to-machine auth, which is what CI/CD pipelines and running services need. The way it works: each service gets a role_id (basically a username — non-sensitive, can be stored openly) and a secret_id (the credential — short-lived, expires after a configurable TTL).

The TTL is the key property. Even if a secret_id leaks from a CI log — and CI logs are the most common place secrets leak from — the attack window is bounded. Ours expire in 1 hour. An attacker who finds an old CI log gets a credential that was valid for an hour, two weeks ago.

# Enable AppRole auth method
vault auth enable approle

# Create a role per service
vault write auth/approle/role/api-service \
    token_policies="api-service-policy" \
    secret_id_ttl=1h \
    token_ttl=30m \
    token_max_ttl=2h \
    bind_secret_id=true

# role_id is non-sensitive — store in CI as a plain variable
vault read auth/approle/role/api-service/role-id

Policy-based access — enforced, not assumed

Before Vault, our billing service and our LLM router service both had access to the same .env file with all secrets in it. The billing service had no business knowing the LLM router API tokens. We just assumed it wouldn't use them. That's not a security model, that's trust without enforcement.

Vault policies make least-privilege enforcement explicit. Each service can only read the secrets on its own path:

# api-service-policy.hcl
path "secret/data/api-service/*" {
  capabilities = ["read", "list"]
}
# Explicit deny — nothing implicit
path "secret/data/billing/*" {
  capabilities = ["deny"]
}
path "secret/data/llm-router/*" {
  capabilities = ["deny"]
}

vault policy write api-service api-service-policy.hcl

I like writing explicit denies even for paths a service would never normally touch. It makes the intent clear in the policy file and catches misconfiguration if the default-deny behavior ever changes.

CI/CD integration — no secrets baked into images

The pattern we use: CI pipeline gets a role_id stored as a plain GitHub Actions variable (not a secret — role_id is non-sensitive by design). At runtime, each CI run requests a fresh secret_id from Vault, exchanges it for a short-lived token, and fetches what it needs. The secret_id expires before the run is even done.

- name: Fetch secrets from Vault
  env:
    VAULT_ADDR: ${{ secrets.VAULT_ADDR }}
    ROLE_ID: ${{ secrets.ROLE_ID }}   # Non-sensitive: like a username
  run: |
    # Fresh secret_id per run — expires in 1h
    SECRET_ID=$(vault write -f \
      auth/approle/role/sonar/secret-id \
      -format=json | jq -r '.data.secret_id')

    # Exchange for a short-lived Vault token (30min)
    VAULT_TOKEN=$(vault write auth/approle/login \
      role_id=$ROLE_ID \
      secret_id=$SECRET_ID \
      -format=json | jq -r '.auth.client_token')

    # Fetch the actual secret
    echo "SONAR_TOKEN=$(VAULT_TOKEN=$VAULT_TOKEN \
      vault kv get -field=token secret/sonar)" >> $GITHUB_ENV

For SonarQube we went one step further: instead of a static token stored in Vault, we configured Vault to generate a unique SonarQube token per CI run that gets automatically revoked when the pipeline finishes. Each run gets credentials that have never existed before. Whether that level of dynamism is worth the setup complexity depends on your risk appetite — for most teams, static tokens in Vault with rotation is plenty.

GDPR Article 32: what on-prem actually gives you

Article 32 requires "appropriate technical and organisational measures" for the security of processing — including encryption key management. If your encryption keys are fetched via an API that routes through US infrastructure, you have a cross-border data transfer under Chapter V that needs to be justified and documented.

On-prem Vault is a clean argument here. When your auditor asks "where does secret retrieval happen and who can access those logs?", the answer is: this server, in this data center, in Germany, and here are the logs. No dependency on a cloud provider's export API, no SCC framework required, no SCCs to renew. The paper trail lives on hardware you control.

This doesn't mean AWS/GCP secrets managers are non-compliant — they're not. But the compliance argument is more complex and requires more documentation. For a small team, owning the simplicity is worth it.

Result: Zero .env files in production · Secret rotation = one Vault write, services pick it up on next token refresh · Every access audited with timestamp + service identity · GDPR Article 32 compliance documented with on-prem audit logs · Vault uptime: 99.8%

What I'd tell you before starting

  • Enable audit logging on day one: vault audit enable file file_path=/var/log/vault_audit.log — this is the log that matters during an incident and during GDPR audits. Don't add it later
  • Write your policies before you write your secrets. It's much easier to enforce least privilege from the start than to retrofit it across a system where every service already has read-all access
  • AppRole is the right auth method for CI/CD from day one. Don't start with long-lived tokens and migrate to AppRole later — the short TTLs are the whole point
  • If GDPR applies to you, document the architecture explicitly in your security documentation. "Vault on-prem in Germany, secrets never route through external APIs" is a sentence your DPO will thank you for having written down
  • LXD for Vault isolation works well — full network namespace, separate filesystem, resource limits. Not as good as a dedicated machine, but far better than running Vault directly on the host

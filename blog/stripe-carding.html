Stopping a Live Stripe Carding Attack — Zero Fraud Post-Deployment

Feb 2026·6 min read·Talha Zafar

I was halfway through lunch when the Stripe notification came in. Then another. Then five more in quick succession. By the time I opened the dashboard, we had 400+ failed charges in two hours — all $0.99, all from the same cluster of Eastern European IPs, all on different card numbers. Our fraud rate had jumped to 12%. This is how we got it to 0.02% in an afternoon.

What a carding attack actually is

Attackers buy databases of stolen card numbers — sometimes millions at a time — and need to figure out which ones are still active before reselling them. The way they do that is find a payment endpoint somewhere on the internet and fire $0.50–$1 test charges. Cards that succeed are "live." Cards that fail get discarded.

Our /api/checkout endpoint was that payment endpoint for someone's testing script. We weren't losing money directly — all the charges were failing — but Stripe monitors your dispute and failure rates, and at 12% we were on track for a warning email threatening account suspension. That gets your attention fast.

What the logs looked like

The pattern was obvious once I sorted by timestamp. Burst of POST requests, perfectly valid JSON payloads, coming from 8–10 rotating IPs. All had the same user-agent string (someone wasn't even trying to hide). Different card numbers every request, but if you looked at the first 6 digits — the BIN prefix that identifies the issuing bank — most of them were from the same 3–4 BIN ranges. That detail turned out to be the key to stopping it.

The other thing I noticed: our payment service had no authentication between it and our API. Anyone who knew the endpoint URL could call it directly without going through the main API gateway. That's how the attacker was hitting us — directly to the payment service, bypassing everything.

Layer 1: Stripe Radar rules (the fastest fix)

Radar is genuinely good. You can write blocking rules that evaluate at Stripe's edge before the charge attempt even hits your server. I added three rules and pushed them immediately:

# Block high-risk country + micro-amount combination
block if: :ip_country: in ('RU', 'UA', 'RO', 'BG', 'BY', 'MD') and :amount_in_cents: < 200

# Block velocity per IP per hour
block if: charge_count_per_ip_hourly > 3

# Review first-time international cards with micro-amounts
review if: :is_new_card: = true and :card_country: != 'PK' and :amount_in_cents: < 500

This killed about 60% of the attack volume at Stripe's edge. The remaining requests were still hitting our server though, just getting blocked before reaching Stripe. We were still burning resources handling them. And more importantly, IP rotation meant the per-IP Radar rule would only slow them down, not stop them.

Layer 2: Redis rate limiting on BIN prefix

The BIN prefix insight was the real solution. A normal customer, even if they try a second card, almost never tries two cards from the same issuing bank within the same hour. But an attacker cycling through a stolen card dump from one bank breach? They'll have dozens of cards with the same BIN prefix.

So we rate-limited on the combination of IP + BIN prefix, not just IP:

async def check_rate_limit(ip: str, card_number: str) -> None:
    bin_prefix = card_number[:6]

    # Per-IP limit
    ip_key = f"rl:ip:{ip}"
    ip_count = await redis.incr(ip_key)
    if ip_count == 1:
        await redis.expire(ip_key, 3600)
    if ip_count > 5:
        raise HTTPException(status_code=429, detail="Rate limit exceeded")

    # Per-IP + BIN limit (catches card cycling from same bank)
    bin_key = f"rl:bin:{ip}:{bin_prefix}"
    bin_count = await redis.incr(bin_key)
    if bin_count == 1:
        await redis.expire(bin_key, 3600)
    if bin_count > 2:
        raise HTTPException(status_code=429, detail="Rate limit exceeded")

The BIN limit of 2 per hour is aggressive, but our actual users — a Pakistani SaaS product at the time — almost never had two failed cards from the same bank in an hour. If you're in a context where that's more common, raise it to 3 or 4, but keep it low. Legitimate card cycling behavior is not normal.

Layer 3: HMAC between services

The direct-to-payment-service access had to go. The fix was HMAC-SHA256 request signing so the payment service only accepts requests that were signed by our own backend — nothing else gets through.

import hmac, hashlib

SHARED_SECRET = settings.INTER_SERVICE_SECRET  # pulled from Vault

def sign_request(payload: bytes) -> str:
    return hmac.new(
        SHARED_SECRET.encode(), payload, hashlib.sha256
    ).hexdigest()

# Calling service adds signature header
headers["X-Internal-Signature"] = sign_request(request_body)

# Payment service verifies before processing
async def verify_internal_sig(request: Request) -> None:
    sig = request.headers.get("X-Internal-Signature", "")
    body = await request.body()
    expected = sign_request(body)
    if not hmac.compare_digest(sig, expected):
        raise HTTPException(status_code=401, detail="Invalid signature")

One thing: use hmac.compare_digest, not ==, for the comparison. The difference is that compare_digest runs in constant time regardless of where the strings differ. A regular == comparison short-circuits on the first mismatched character, which leaks timing information — an attacker can guess the correct signature one character at a time by measuring how long the comparison takes. This is a real attack that works in practice, not a theoretical concern.

End state: Fraud rate 12% → 0.02% · Zero fraudulent transactions after deployment · No legitimate customers got blocked · Stripe fraud warning resolved · Total time: under 4 hours from detection to green dashboard

What I'd do differently

Honestly, most of this should have been in place from day one. The Radar velocity rules take 10 minutes to configure. The HMAC signing between services is maybe a day of work. Neither of them is the kind of thing you think about when you're moving fast building features — until you're staring at 400 failed charges over lunch.

  • Add Radar velocity rules before you launch, not after an incident. Block IP-based velocity from day one even if the limits are generous
  • Rate limit on BIN prefix, not just IP. Attackers rotate IPs; they can't rotate away from a stolen card dump's BIN
  • Internal services should require auth. If it touches payments or user data, it should verify the caller — no exceptions
  • Enable Stripe webhook logging and keep it. The event stream (timestamps, card fingerprints, IPs) is the audit trail you'll want during and after an incident

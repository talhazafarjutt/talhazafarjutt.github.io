Stopping a Live Stripe Carding Attack — Zero Fraud Post-Deployment

Feb 2026·6 min read·Talha Zafar

400+ failed charges in two hours, all $0.99, all from Eastern Europe. Our fraud rate hit 12%. We went to 0.02% in under 4 hours by layering three defences: Stripe Radar rules, Redis BIN-prefix rate limiting, and HMAC inter-service authentication.

What is a Carding Attack?

Attackers buy lists of stolen card numbers and run automated scripts to test them in small amounts — usually $0.50–$1.00 — against real merchant endpoints. Cards that succeed are "live" and sold at a premium on dark web marketplaces. You only notice when Stripe sends a warning email about elevated dispute rates or your payment processor threatens account termination.

How We Found Out

Our Stripe dashboard showed 400+ failed charges in a 2-hour window: all $0.99, multiple IPs from Eastern Europe, all hitting the same /api/checkout endpoint, all with different card numbers but similar BIN prefixes. Classic carding attack signature.

The pattern in logs: burst of POST requests, all with valid JSON payloads, all failing at the Stripe authorization step. Our API was being used as a card-testing proxy.

Layer 1: Stripe Radar Rules

Radar's rule DSL lets you block or review charges based on risk signals evaluated at Stripe's edge — before the charge attempt even reaches your server. We added three rules immediately:

# Block high-risk country + micro-amount combination
block if: :ip_country: in ('RU', 'UA', 'RO', 'BG', 'BY', 'MD') and :amount_in_cents: < 200

# Block velocity per IP per hour
block if: charge_count_per_ip_hourly > 3

# Review first-time international cards with micro-amounts
review if: :is_new_card: = true and :card_country: != 'PK' and :amount_in_cents: < 500

This cut ~60% of attack volume at Stripe's edge. The remaining attacks were hitting our API server but getting blocked before reaching Stripe — still a problem because we were burning server resources processing them.

Layer 2: Redis Rate Limiting Per IP + BIN Prefix

We needed to stop the requests reaching our API at all. A token bucket per IP address is the obvious approach, but attackers can rotate IPs through proxies. The more effective signal: BIN prefix (first 6 digits of a card number, which identifies the issuing bank). Legitimate users rarely try more than one card from the same bank in an hour.

async def check_rate_limit(ip: str, card_number: str) -> None:
    bin_prefix = card_number[:6]

    # Per-IP limit
    ip_key = f"rl:ip:{ip}"
    ip_count = await redis.incr(ip_key)
    if ip_count == 1:
        await redis.expire(ip_key, 3600)
    if ip_count > 5:
        raise HTTPException(status_code=429, detail="Rate limit exceeded")

    # Per-IP + BIN limit (catches card cycling from same bank)
    bin_key = f"rl:bin:{ip}:{bin_prefix}"
    bin_count = await redis.incr(bin_key)
    if bin_count == 1:
        await redis.expire(bin_key, 3600)
    if bin_count > 2:
        raise HTTPException(status_code=429, detail="Rate limit exceeded")

The BIN-prefix limit was the key insight. Attackers who rotated cards from the same issuing bank (a common pattern — stolen card dumps often come from single bank breaches) got blocked even with different card numbers.

Layer 3: HMAC Inter-Service Authentication

Our payment service was only supposed to be called from our own backend API. But there was no enforcement — any client that knew the endpoint could call it directly. We added HMAC-SHA256 request signing between services so the payment service rejects anything that didn't come from a verified internal caller.

import hmac, hashlib

SHARED_SECRET = settings.INTER_SERVICE_SECRET  # from Vault

def sign_request(payload: bytes) -> str:
    return hmac.new(
        SHARED_SECRET.encode(), payload, hashlib.sha256
    ).hexdigest()

# Calling service adds signature header
headers["X-Internal-Signature"] = sign_request(request_body)

# Payment service verifies before processing
async def verify_internal_sig(request: Request) -> None:
    sig = request.headers.get("X-Internal-Signature", "")
    body = await request.body()
    expected = sign_request(body)
    if not hmac.compare_digest(sig, expected):
        raise HTTPException(status_code=401, detail="Invalid signature")

Using hmac.compare_digest instead of == prevents timing attacks — an attacker can't guess the signature one byte at a time by measuring response times.

Result: Fraud rate 12% → 0.02% · Zero fraudulent transactions after deployment · No legitimate customer blocked · Full mitigation deployed in under 4 hours · Stripe fraud warning resolved

What We Learned

  • Configure Radar rules proactively — reactive setup during an active attack is stressful; add baseline velocity rules from day one
  • Rate limit on BIN prefix (first 6 digits), not just full card — attackers cycle cards from the same issuing bank to evade per-card limits
  • HMAC between your own services is cheap to add and eliminates an entire attack surface — don't leave internal APIs open
  • Keep a Stripe webhook log — the audit trail (event timestamps, card fingerprints, IP addresses) is invaluable for post-incident analysis and Stripe dispute resolution
  • Use hmac.compare_digest, never == for signature comparison — prevents timing side-channel attacks

gRPC vs REST in Production — How We Cut Internal API Overhead by 40%

Jan 2026·7 min read·Talha Zafar

Six microservices. All REST over HTTP/1.1. A single agent task triggered 5 synchronous cross-service calls, each with JSON serialisation overhead and connection setup. Here's what we measured, how we migrated with zero downtime, and when gRPC is — and isn't — the right call.

The Setup

The AI agent platform had 6 independent services: API Gateway, Agent Orchestrator, Task Queue, LLM Router, User Service, and Billing. All communication was REST over HTTP/1.1.

A single agent task execution path: Gateway → Orchestrator → LLM Router → Task Queue → back to Orchestrator. That's five synchronous round-trips per user request, each with:

  • JSON serialisation overhead (Python's json.dumps / json.loads on every hop)
  • HTTP/1.1 connection overhead (new connection per request without keep-alive tuning)
  • No native streaming (LLM token streaming required separate SSE endpoint)
  • No typed contracts (field naming inconsistencies caught only at runtime)

Why gRPC?

  • Protocol Buffers: binary serialisation — ~3–10× smaller payloads than JSON, faster to parse
  • HTTP/2: connection multiplexing — one persistent connection per service pair, no per-request TCP handshake
  • Server-side streaming: LLM token streaming over gRPC is natural with stream return types; over REST it required a separate SSE implementation
  • Typed contracts: .proto files as the source of truth — schema mismatches caught at compile time, not production

The .proto Contract

syntax = "proto3";

service AgentService {
  rpc ExecuteTask(TaskRequest) returns (TaskResponse);
  rpc StreamOutput(TaskRequest) returns (stream OutputChunk);
  rpc GetStatus(StatusRequest) returns (StatusResponse);
}

message TaskRequest {
  string task_id    = 1;
  string agent_id   = 2;
  string prompt     = 3;
  int32  max_tokens = 4;
  map<string, string> metadata = 5;
}

message OutputChunk {
  string content  = 1;
  bool   is_final = 2;
  int32  token_count = 3;
}

The .proto file becomes the API contract. Any service that needs to call AgentService generates its client from this file — no documentation drift, no "what does this field actually mean?" Slack threads.

Migration Strategy — No Big-Bang Rewrite

We didn't rewrite everything simultaneously. The strategy: add gRPC as a second transport alongside REST, migrate service-to-service calls first, leave the public API as REST. Each service remained testable via REST throughout the transition.

Migration order mattered: we started with the highest-frequency call path (Orchestrator → LLM Router), measured the improvement, then proceeded. Total migration time: two weeks, zero downtime, no emergency rollbacks.

Python gRPC with asyncio

import grpc
from grpc.aio import insecure_channel, Channel
from proto.agent_pb2_grpc import AgentServiceStub
from proto.agent_pb2 import TaskRequest

class AgentClient:
    def __init__(self, address: str):
        self._channel = insecure_channel(address)
        self._stub = AgentServiceStub(self._channel)

    async def execute(self, task_id: str, prompt: str) -> TaskResponse:
        req = TaskRequest(task_id=task_id, prompt=prompt, max_tokens=2048)
        return await self._stub.ExecuteTask(req, timeout=30)

    async def stream(self, task_id: str, prompt: str):
        req = TaskRequest(task_id=task_id, prompt=prompt)
        async for chunk in self._stub.StreamOutput(req):
            yield chunk.content
            if chunk.is_final:
                break

When NOT to Use gRPC

  • Public-facing APIs: browsers can't use gRPC natively without grpc-web, which adds a proxy layer and complexity
  • Simple infrequent CRUD: the overhead of proto compilation and generated code isn't worth it for endpoints called 10 times/day
  • Teams unfamiliar with proto tooling: debugging binary protocol traffic is significantly harder than JSON — invest in Postman/grpcurl tooling first
  • External partner integrations: REST with OpenAPI documentation is more universally accessible to third-party developers

Takeaway

For service-to-service calls within the same infrastructure, gRPC is almost always the right choice. The combination of smaller payloads, HTTP/2 multiplexing, and typed contracts pays dividends at any scale. Keep REST for your public API and external integrations where client diversity matters.

The migration path doesn't have to be all-or-nothing — running REST and gRPC transports in parallel during migration is clean and reversible.

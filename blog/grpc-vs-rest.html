gRPC vs REST in Production — How We Cut Internal API Overhead by 40%

Jan 2026·7 min read·Talha Zafar

About eight months into building our AI agent platform, I got annoyed enough at latency on the internal service calls that I pulled up a quick benchmark. The numbers were bad enough that we spent the next two weeks migrating everything to gRPC. Here's what we measured, how the migration went, and — equally important — what I'd still keep on REST.

The setup we were working with

Six services: API Gateway, Agent Orchestrator, Task Queue, LLM Router, User Service, and Billing. All FastAPI. All communicating over REST/HTTP/1.1 with JSON. Standard stuff, and it was fine at low load.

The problem showed up when we started tracing individual agent task execution. A single user request would trigger five synchronous cross-service calls: Gateway → Orchestrator → LLM Router → Task Queue → back to Orchestrator. Each hop was adding JSON serialization, a new TCP connection (we hadn't tuned keep-alive), and the general overhead of an HTTP/1.1 request. Under load, the P95 end-to-end latency on agent tasks was creeping toward 200ms just in internal calls — before LLM inference was even included.

There was also a more annoying problem: field naming drift. With no schema enforcement between services, we'd had two separate incidents where one service changed a JSON field name and broke a downstream consumer that nobody caught until it was in production. Not a big deal individually, but a pattern I wanted to kill.

Why gRPC addresses these specifically

I've seen people recommend gRPC based on benchmarks from optimized benchmark environments that don't reflect real usage. The concrete advantages that applied to our situation:

  • Protocol Buffers are binary and typed. No JSON parsing overhead, no field naming ambiguity — the .proto file is the contract and it's enforced at compile time
  • HTTP/2 multiplexing means one persistent connection per service pair. Our five round-trips per request were each paying TCP handshake overhead because we hadn't tuned HTTP keep-alive. HTTP/2 makes this a non-issue by default
  • Native server-side streaming — we had a separate SSE endpoint for LLM token streaming over REST. gRPC's stream return type made that the default, not a special case

The .proto file as the contract

This is honestly one of the underrated parts. Once you write the .proto, every service that talks to this one generates its client from it. No more "wait what type is that field" conversations. No more discovering field renames at runtime.

syntax = "proto3";

service AgentService {
  rpc ExecuteTask(TaskRequest) returns (TaskResponse);
  rpc StreamOutput(TaskRequest) returns (stream OutputChunk);
  rpc GetStatus(StatusRequest) returns (StatusResponse);
}

message TaskRequest {
  string task_id    = 1;
  string agent_id   = 2;
  string prompt     = 3;
  int32  max_tokens = 4;
  map<string, string> metadata = 5;
}

message OutputChunk {
  string content     = 1;
  bool   is_final    = 2;
  int32  token_count = 3;
}

The field numbers (1, 2, 3...) are how Protobuf maintains backward compatibility — you can add new fields without breaking old clients as long as you don't reuse field numbers. One thing that tripped us up early: renaming a field in the .proto is fine (Protobuf ignores field names in the wire format), but removing a field and reusing its number breaks everything. Add fields, never reuse numbers.

Migration strategy — no big bang rewrite

The worst thing you can do is try to migrate everything at once. We ran gRPC and REST in parallel for each service during the transition. Each service got a gRPC server added alongside its existing FastAPI REST server. Internal service-to-service calls migrated to gRPC first; the public API stayed REST throughout.

Migration order: highest-call-frequency path first. We started with Orchestrator → LLM Router because that was the hottest path. Measured the improvement, confirmed nothing broke, then moved to the next one. Total time: two weeks. Zero downtime. Zero emergency rollbacks.

Python gRPC with asyncio

One thing worth knowing: the Python gRPC library (grpcio) has async support through grpc.aio, but it's a separate import path from the sync version. If you're mixing sync and async code you can end up with confusing behavior. Stick to grpc.aio throughout if you're on asyncio:

import grpc
from grpc.aio import insecure_channel
from proto.agent_pb2_grpc import AgentServiceStub
from proto.agent_pb2 import TaskRequest

class AgentClient:
    def __init__(self, address: str):
        self._channel = insecure_channel(address)
        self._stub = AgentServiceStub(self._channel)

    async def execute(self, task_id: str, prompt: str) -> TaskResponse:
        req = TaskRequest(task_id=task_id, prompt=prompt, max_tokens=2048)
        return await self._stub.ExecuteTask(req, timeout=30)

    async def stream(self, task_id: str, prompt: str):
        req = TaskRequest(task_id=task_id, prompt=prompt)
        async for chunk in self._stub.StreamOutput(req):
            yield chunk.content
            if chunk.is_final:
                break

Reuse the channel. Creating a new gRPC channel per request completely defeats the HTTP/2 multiplexing benefit — you'd be doing TCP handshakes again on every call. Create one channel per service address at startup and share it.

The benchmark numbers

We measured the Orchestrator → LLM Router path because it was both the highest frequency and the most payload-heavy. 5000 requests each way, identical payloads:

# Before (REST/JSON): 5000 requests, avg payload 1.4KB
avg_latency: 12.3ms | p95: 18.1ms | p99: 31.4ms
payload_size: 1,420 bytes avg

# After (gRPC/Protobuf): 5000 requests, avg payload 340B
avg_latency: 7.1ms  | p95: 10.2ms | p99: 14.8ms
payload_size: 342 bytes avg

# Delta
latency_reduction: 42%
payload_reduction: 76%
p95_improvement:   44%

The payload reduction was more dramatic than I expected. 1.4KB of JSON serializing to 342 bytes of Protobuf is a 76% reduction. At 10,000 agent tasks per day across five hops, that's meaningful bandwidth saved — but more importantly, it's serialization CPU time you're not burning on every request.

Numbers: Internal latency avg 12ms → 7ms (−42%) · Payload 1.4KB → 340 bytes (−76%) · P95 end-to-end agent task 180ms → 108ms · Connection overhead eliminated via HTTP/2 multiplexing

When gRPC is the wrong choice

I want to be clear about this because I've seen teams try to gRPC everything and create more problems than they solve:

  • Public-facing APIs: browsers can't speak gRPC natively. grpc-web exists but adds a proxy layer and complexity. Keep your public API REST with OpenAPI docs
  • Low-frequency CRUD: an endpoint called 20 times a day doesn't need the proto compilation overhead and generated client code. REST is fine
  • External partner integrations: REST + OpenAPI is the universal language of third-party API consumers. Don't make their lives harder
  • Debugging: binary protocol traffic is much harder to inspect than JSON. You need grpcurl or similar tooling. Make sure your team has it set up before you're debugging a production issue at midnight

The short version

For internal service-to-service calls in a microservices architecture: gRPC is almost always worth it. The combination of typed contracts, binary serialization, and HTTP/2 multiplexing adds up. For anything external-facing or low-frequency: REST is fine and requires less infrastructure overhead.

The migration doesn't have to be all-or-nothing. Running both transports in parallel during the switch is clean, testable, and reversible. Start with your hottest call path, measure, then proceed.

Running LLM Inference on 4× Tesla V100 — Bare Metal to Production

Mar 2026·10 min read·Talha Zafar

The first time I tried to run Qwen2.5-Coder-32B on our V100s, it just crashed. No useful error — just a silent OOM. Three hours of debugging later I realized I'd set gpu_memory_utilization to 0.95. Don't do that. This is the writeup I wish I'd had before starting.

Why we didn't just use AWS

The honest answer: cost. AWS p3.8xlarge gives you 4× V100 at $12.24/hr. Run that 24/7 and you're looking at $8,800 a month. We already owned the hardware — four Tesla V100 PCIe cards in a colocated bare-metal server in Germany. Once you factor in the colo fee and power, it came out to roughly $180/month. The ROI math doesn't take a spreadsheet.

The less obvious reason: data residency. We're building AI tooling for European customers, and having LLM inference requests route through US AWS infrastructure was a compliance conversation nobody wanted to have. On-prem meant the data never left the rack.

Getting CUDA to behave on Ubuntu 22.04

I started with Ubuntu 24.04 because it was newer. That was a mistake. The nvidia-container-toolkit packages were lagging behind and I spent most of a day chasing dependency conflicts. Wiped it, installed 22.04 LTS, and everything just worked. Sometimes boring is correct.

distribution=$(. /etc/os-release; echo $ID$VERSION_ID)
curl -s -L https://nvidia.github.io/libnvidia-container/gpgkey | apt-key add -
apt install nvidia-cuda-toolkit nvidia-container-toolkit
systemctl restart docker
# Check that all 4 GPUs actually show up
nvidia-smi

That last command matters more than it seems. I had a card that would show up in lspci but not in nvidia-smi due to a PCIe slot power issue. Better to find that out now than after you've deployed vLLM and are wondering why tensor_parallel_size=4 keeps failing.

K3s — Kubernetes without the headache

Full Kubernetes on a single bare-metal node felt like renting a bus to drive to the corner shop. K3s gives you the same API surface — same manifests, same kubectl, same RBAC — at a fraction of the memory overhead. Install takes about 30 seconds.

The components I actually needed:

  • nginx Ingress Controller — for external traffic routing, TLS termination via cert-manager
  • NVIDIA Device Plugin — this is the important one. Without it, K3s has no idea the GPUs exist as schedulable resources
  • CoreDNS — internal service discovery. I also needed it to resolve LXD container hostnames from within K3s pods
curl -sfL https://get.k3s.io | sh -
kubectl apply -f https://raw.githubusercontent.com/NVIDIA/k8s-device-plugin/main/nvidia-device-plugin.yml
# Wait a minute, then verify the node sees the GPUs
kubectl describe nodes | grep nvidia.com/gpu

If that last command shows nvidia.com/gpu: 4, you're good. If it shows 0 or nothing, the device plugin DaemonSet probably isn't running — check kubectl get pods -n kube-system.

vLLM + Qwen2.5-Coder-32B: the memory math

Qwen2.5-Coder-32B needs about 60–64GB VRAM in BF16 precision. We have 4× 16GB = 64GB total, which sounds fine until you realize the KV cache also needs VRAM. This is where the 0.95 setting killed me.

The gpu_memory_utilization parameter controls how much of each GPU's VRAM vLLM reserves for the KV cache pool at startup. At 0.95, it was trying to grab 15.2GB per card for the KV cache on top of the model weights. That's more VRAM than physically exists. The sweet spot I landed on was 0.91:

from vllm import LLM, SamplingParams

llm = LLM(
    model="Qwen/Qwen2.5-Coder-32B-Instruct",
    tensor_parallel_size=4,
    gpu_memory_utilization=0.91,
    max_model_len=8192,
    dtype="bfloat16"
)

One thing worth knowing about tensor_parallel_size on V100 PCIe: these cards don't have NVLink. Communication between GPUs goes through the CPU's PCIe bus, which is slower than NVLink on data center V100s. For a 32B model this is fine — the bottleneck is still compute, not interconnect. But if you try to run a 70B+ model across PCIe V100s and wonder why throughput is miserable, that's why.

The FastAPI load balancer

Each GPU runs its own vLLM process as an OpenAI-compatible server on a separate port. I wrote a small FastAPI router to distribute requests across them. The naive approach — just round-robin — had one problem: after a pod restart or node reboot, vLLM takes 3–4 minutes to load the model back into VRAM. During that window, round-robin was sending requests to a worker that wasn't ready and silently returning errors.

The fix was a readiness check before routing:

from itertools import cycle
import httpx
from fastapi import FastAPI, Request, Response, Depends

app = FastAPI()
worker_pool = cycle([
    "http://llm-worker-0:8000",
    "http://llm-worker-1:8000",
    "http://llm-worker-2:8000",
    "http://llm-worker-3:8000",
])

@app.post("/v1/completions")
async def complete(req: Request, api_key: str = Depends(verify_key)):
    target = next(worker_pool)
    async with httpx.AsyncClient(timeout=120) as client:
        r = await client.post(
            f"{target}/v1/completions",
            content=await req.body(),
            headers={"Content-Type": "application/json"}
        )
    return Response(content=r.content, media_type="application/json")

The 120-second timeout is intentional. Long-context requests on V100s can take a while, and the default httpx timeout of 5 seconds will bite you on anything over ~500 tokens output.

nginx rate limiting — keep the API from being abused

The team was using this internally but I wanted API key auth and per-key rate limits in place before thinking about any external access. nginx handles this cleanly with its limit_req_zone directive:

upstream llm_upstream {
    server llm-router:8080;
}
limit_req_zone $http_x_api_key zone=per_key:10m rate=30r/m;

server {
    location /v1/ {
        auth_request     /auth;
        proxy_pass       http://llm_upstream;
        limit_req        zone=per_key burst=10 nodelay;
        proxy_read_timeout 120s;
        proxy_buffering  off;
    }
}

proxy_buffering off matters here — without it, nginx buffers the entire response before sending it to the client, which breaks streaming completions.

LXD for developer containers on the same server

Alongside the K3s production workload, the same physical server runs 8 LXD containers — one per developer, plus a few shared services. LXD gives near-native performance and full filesystem isolation without VM overhead. The networking setup took the longest: getting MASQUERADE NAT working on the bond0 interface so containers had internet access was a solid afternoon of reading iptables docs.

# NAT for LXD container internet access
iptables -t nat -A POSTROUTING -s 10.x.x.0/24 -o bond0 -j MASQUERADE
# Make it survive reboots
echo "iptables-save > /etc/iptables/rules.v4" >> /etc/rc.local

VS Code Remote tunnels for each engineer connect through here. It works well enough that nobody asks me to SSH manually anymore, which I take as a success metric.

Where we ended up: 10,000+ daily completions to a 5-engineer team · Cold start ~3.5 min (acceptable) · P95 token generation ~400ms for 256-token output · ~$180/month vs ~$8,800 cloud equivalent · 99.8% uptime

Things I'd tell myself at the start

  • Don't use Ubuntu 24.04 for this yet — 22.04 LTS has better-tested nvidia-container-toolkit packages
  • Set gpu_memory_utilization to 0.90–0.92, not higher. The headroom prevents OOM crashes when long-context requests spike the KV cache
  • V100 PCIe has no NVLink — tensor parallel communication goes through CPU PCIe. Fine for 32B, rough for 70B+
  • Add a readiness probe before routing to any vLLM worker — model load takes 3–4 minutes and a worker will silently fail during that window
  • Set proxy_buffering off in nginx if you want streaming to actually stream

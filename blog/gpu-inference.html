Running LLM Inference on 4× Tesla V100 — Bare Metal to Production

Mar 2026·10 min read·Talha Zafar

AWS p3.8xlarge (4× V100, 16GB each) costs $12.24/hour — $8,800/month for 24/7 availability. We owned the hardware. Colocation + electricity brought the effective monthly cost to ~$180. Here's the full stack that serves our LLM inference platform in production.

The Why: Cloud GPUs Are Expensive

AWS p3.8xlarge gives you 4× V100 at $12.24/hr. That's $8,800/month for always-on availability. Owning bare metal with hosted colocation cuts that to electricity and depreciation — in our case, ~$180/month effective cost. The ROI calculation was easy.

OS & CUDA Setup

Ubuntu 22.04 LTS — not 24.04. CUDA driver stability and nvidia-container-toolkit package availability were better tested on 22.04 at the time of setup. We installed CUDA 12.x and the NVIDIA Container Toolkit to allow Docker and K3s pods to access the physical GPUs.

distribution=$(. /etc/os-release; echo $ID$VERSION_ID)
curl -s -L https://nvidia.github.io/libnvidia-container/gpgkey | apt-key add -
apt install nvidia-cuda-toolkit nvidia-container-toolkit
systemctl restart docker
# Verify GPU visibility
nvidia-smi

K3s: Lightweight Kubernetes on One Server

Full Kubernetes felt like overkill for a single physical node. K3s runs the same Kubernetes API surface at a fraction of the resource overhead. We deployed:

  • nginx Ingress Controller — external traffic routing with TLS termination
  • cert-manager + Let's Encrypt — automatic TLS, zero manual certificate management
  • CoreDNS — internal service discovery across LXD containers and K3s pods
  • NVIDIA Device Plugin — exposes nvidia.com/gpu as a schedulable Kubernetes resource

Each vLLM worker pod requests exactly nvidia.com/gpu: 1 in its resource spec, so K3s handles GPU allocation without any manual assignment.

# k3s-install with GPU support
curl -sfL https://get.k3s.io | sh -
# Apply NVIDIA device plugin
kubectl apply -f https://raw.githubusercontent.com/NVIDIA/k8s-device-plugin/main/nvidia-device-plugin.yml
# Verify GPU scheduling works
kubectl describe nodes | grep nvidia.com/gpu

vLLM + Qwen2.5-Coder-32B

Qwen2.5-Coder-32B needs ~60GB VRAM in BF16 precision. With 4× 16GB = 64GB total, there's just enough headroom for the KV cache. tensor_parallel_size=4 splits the model across all four V100s using PCIe communication.

from vllm import LLM, SamplingParams

llm = LLM(
    model="Qwen/Qwen2.5-Coder-32B-Instruct",
    tensor_parallel_size=4,
    gpu_memory_utilization=0.91,
    max_model_len=8192,
    dtype="bfloat16"
)

Keep gpu_memory_utilization at 0.90–0.92. Going higher causes OOM crashes during KV cache spikes on long-context requests — we learned this the hard way after hitting 0.95.

FastAPI Load-Balancing Router

Each GPU worker runs as an independent vLLM OpenAI-compatible server on a different port. A FastAPI router round-robins across them with health checks — no traffic goes to a worker still loading the model (cold start is 3–4 minutes).

from itertools import cycle
import httpx
from fastapi import FastAPI, Request, Response, Depends

app = FastAPI()
worker_pool = cycle([
    "http://llm-worker-0:8000",
    "http://llm-worker-1:8000",
    "http://llm-worker-2:8000",
    "http://llm-worker-3:8000",
])

@app.post("/v1/completions")
async def complete(req: Request, api_key: str = Depends(verify_key)):
    target = next(worker_pool)
    async with httpx.AsyncClient(timeout=120) as client:
        r = await client.post(
            f"{target}/v1/completions",
            content=await req.body(),
            headers={"Content-Type": "application/json"}
        )
    return Response(content=r.content, media_type="application/json")

nginx: API Key Auth + Rate Limiting

upstream llm_upstream {
    server llm-router:8080;
}
limit_req_zone $http_x_api_key zone=per_key:10m rate=30r/m;

server {
    location /v1/ {
        auth_request     /auth;
        proxy_pass       http://llm_upstream;
        limit_req        zone=per_key burst=10 nodelay;
        proxy_read_timeout 120s;
        proxy_buffering  off;
    }
}

LXD Developer Containers

The same bare-metal server also runs 8+ LXD containers for individual developers. LXD provides near-native performance (unlike Docker) with full filesystem isolation. We configured MASQUERADE NAT rules on the bond0 interface so containers have internet access, and VS Code Remote tunnel connectivity for all 5 engineers.

# NAT for LXD container internet access
iptables -t nat -A POSTROUTING -s 10.x.x.0/24 -o bond0 -j MASQUERADE
# Make persistent
echo "iptables-save > /etc/iptables/rules.v4" >> /etc/rc.local
Results: 10,000+ daily completions served to a 5-engineer team · Cold start ~3.5 min · P95 token generation ~400ms for 256-token output · Monthly infra cost ~$180 vs ~$8,800 cloud equivalent · 99.8% uptime maintained

Key Takeaways

  • K3s + NVIDIA device plugin works on single-node bare metal with zero special configuration beyond the device plugin DaemonSet
  • tensor_parallel_size must match the number of physical GPUs exactly for V100 PCIe topology — NVLink isn't available on V100 PCIe cards
  • Always health-check poll workers before routing — model load takes 3–4 minutes and silently returns errors without explicit readiness checks
  • LXD containers for dev isolation + K3s for production on the same host works well; use CoreDNS to bridge the two networking domains
  • Set gpu_memory_utilization ≤ 0.92 — the remaining headroom prevents OOM on burst KV cache allocations
